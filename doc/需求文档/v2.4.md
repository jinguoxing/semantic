# BizSemantic v2.4

## 功能与改进点汇总（基于 v2.3 演进）

---

## 一、v2.4 的整体定位（先定调）

### v2.3 已解决什么？

* Bottom-up 语义理解闭环（规则 + AI + 人工确认）
* 业务对象（L1）与映射（L3）的稳定资产化
* 术语、标签、Run Snapshot、评分模板等“企业级可信”能力

### v2.4 要解决什么？

> **解决“没有数据也能开始、数据来了不推翻”的问题**

一句话总结 v2.4：

> **让 BizSemantic 成为“业务认知 → 数据验证 → 语义资产”的统一入口**

---

## 二、v2.4 新增的核心能力（新增为主）

---

## 1️⃣ 业务对象草模（Draft Object）能力（Top-down 起点）

### 目标

* 支持“没有库表先建对象”
* 防止退回传统概念建模

### 新功能

* 创建 **业务对象草模（Draft）**
* 基于 L0（对象 / 过程 / 结果）的事实梳理流程
* Draft 状态明确标识（不可直接生效）

### 关键约束（写进 PRD）

* Draft 阶段：

  * ❌ 不允许绑定系统/表/字段
  * ❌ 不允许定义完整字段模型
  * ❌ 不允许下发为分析标准

---

## 2️⃣ Draft → Active 状态机（对象生命周期）

### 新增状态

* DRAFT
* REVIEW
* ACTIVE
* DEPRECATED
* （可选）PENDING_DATA / MATCHING / ARCHIVED

### 新功能

* 对象审核流（Review Checklist）
* 发布对象（Active）显式动作
* 对象弃用 / 恢复 / 归档

### 新数据结构

* `object_review_task`
* `object_match_suggestion`

---

## 3️⃣ Draft × Bottom-up 自动匹配机制（关键）

### 目标

* 数据接入后，自动验证 Draft 假设

### 新功能

* Draft 对象与新扫描表的自动匹配
* 输出三类建议：

  * STRONG_MATCH
  * PARTIAL_MATCH
  * CONFLICT

### UI 改进

* “哪些数据支持了你的设想”
* “哪些地方与你的定义不一致”

---

## 4️⃣ 智能标签推荐（Decorative Semantics）

### 新增能力

* 在语义理解阶段并行生成标签建议
* 标签类型：

  * domain / shape / usage / risk / tech

### 技术实现

* Rule 派生标签（强一致）
* AI 推荐标签（弱、可解释）

### 重要护栏

* ❌ 标签不参与 Gate / Score / Upgrade
* ❌ 标签不影响对象类型

---

## 5️⃣ 术语候选在语义理解阶段提前生成（但不生效）

### 新能力

* 在语义理解阶段生成：

  * 标准名候选
  * 别名候选
  * 术语复用提示

### 生效规则

* 仅在 REVIEW → ACTIVE 时写入术语资产
* 术语必须绑定对象，不可独立存在

---

## 6️⃣ Top-down × Bottom-up 的统一时序规范（产品级）

### 明确冻结顺序

```
元数据扫描
→ Bottom-up 语义理解
→ 建议集合（对象 / 字段 / 标签 / 术语候选）
→ 引入 Top-down（已有对象 / 术语 / 场景）
→ 对比 & 冲突解释
→ 人工确认
→ 资产入库
```

### 新增内容

* PRD 中增加「设计红线说明」
* 禁止 Top-down 在识别前介入

---

## 三、v2.4 的关键改进点（在 v2.3 基础上加强）

---

## 7️⃣ 语义理解阶段的“建议边界”强化

### 改进点

* 明确区分：

  * Suggestion（建议）
  * Decision（裁决）
* UI 强化“未生效”状态

---

## 8️⃣ 对象、术语、标签的认知负担防护设计

### 改进点

* UI 永远以“业务对象”为中心
* 术语仅以“别名 / 也称”呈现
* 标签仅允许：

  * 展示
  * 筛选
  * 忽略

### 禁止

* 独立术语管理页（默认）
* 独立标签管理页（默认）

---

## 9️⃣ Run Snapshot 覆盖 Draft 阶段

### 新增

* Draft 匹配过程也生成 Run Snapshot
* 支持：

  * Draft 定义 vs 数据现实 对比
  * 不同时间数据接入的验证差异

---

## 10️⃣ 对象发布与映射解耦（增强）

### 改进点

* 允许：

  * 先发布 L1（对象）
  * 后补 L3（映射）
* 显式提示“映射未完成”的对象

---

## 四、v2.4 MVP 实施优先级（建议）

### P0（必须）

1. 业务对象 Draft 创建流程（L0 → L1-Draft）
2. Draft → Review → Active 状态机
3. Draft × 数据匹配建议
4. 智能标签（只读）

### P1（强烈建议）

5. 术语候选提前生成
6. Review Checklist
7. Run Snapshot 覆盖 Draft

### P2（可延后）

8. Draft 对象关系弱建模
9. 批量 Draft 治理视图
10. 标签运营统计

---

## 五、v2.4 成功的标志（验收标准）

你可以用这 5 条来验收 v2.4 是否成功：

1. **没有库表，业务也能正确开始**
2. **数据到来后，不需要推翻已有对象**
3. **任何语义资产都能说清“为什么这么定”**
4. **AI 输出永远是建议，而不是事实**
5. **业务对象能跨系统长期复用**
