# 一、这个 Frame 在产品里的定位（先统一认知）

**所在路径**

```
数据语义治理
  > 业务梳理
    > 场景详情
      > 场景配置（Template & Capabilities）【新增】
```

**它解决的问题**

* 明确：**这个场景“是什么类型的业务”**
* 决定：**后续识别 / 建模 / 校验的强制项**
* 是 Top-down 的**“第一道闸门”**

👉 这是你现在页面里**最缺、但价值最高的一个 Frame**。

---

# 二、Figma Frame 总览

### Frame 名称

```
SceneTemplateConfig
```

### Frame 布局建议

* **宽度**：与当前「业务场景描述」同宽
* **位置**：

  * 推荐放在 **业务场景描述上方**
  * 或作为一个 **可折叠 Section**

---

# 三、Frame 结构拆解（从上到下）

---

## Section 1：场景类型选择（Scene Type）

### 🎯 目的

* 让系统明确：这是政务？企业？哪一类？
* 这是所有后续能力开关的**母选项**

---

### Component：SceneTypeSelector

**展示形式**

* 单选卡片（Radio Card）
* 带图标 + 简短说明

#### 字段结构

```
scene_type: enum
```

#### 选项（初版就够用）

* 🏛 政务 · 一件事

  * 描述：跨部门、多事项整合的政务服务场景
* 🏢 企业 · HR

  * 描述：员工全生命周期管理场景
* 🏭 企业 · 供应链

  * 描述：订单、采购、履约类业务场景
* 🧩 通用业务场景

  * 描述：尚未分类或自定义业务

#### 交互规则

* 必选
* 切换时：

  * 下方能力配置 **实时变化**
  * 已识别结果不清空，但标记“需重新校验”

---

## Section 2：通用能力（Base Capabilities）

### 🎯 目的

* 明确：**哪些是“所有场景都必须具备”的建模骨架**
* 这些能力 **不可关闭**

---

### Component：BaseCapabilityList

**展示形式**

* Checklist（已勾选 + 锁定）

#### 固定能力项

```
☑ 核心业务对象（BO）        🔒
☑ 状态机（State Machine）   🔒
☑ 业务行为（Actions）       🔒
☑ 角色与责任（Roles）       🔒
☑ 材料与数据（Artifacts）   🔒
☑ 业务规则（Rules）         🔒
```

#### 说明文案（很重要）

> 这些是业务语义建模的基础能力，所有场景默认启用

---

## Section 3：增强能力配置（Capability Modules）

> **这是整个 Frame 的核心价值**

---

### Component：CapabilityModuleGroup

#### 布局

* 按「模块卡片」分组
* 每个模块可展开 / 折叠

---

### 3.1 政务增强模块（Government Enhancement）

#### 显示条件

* scene_type = 政务 · 一件事

---

#### Module Card：GovernmentEnhancement

**卡片头**

```
🏛 政务增强能力
```

**能力项（Checkbox）**

```
☑ 多部门核验
  - 公安 / 民政 / 卫健 / 住建等

☑ 法定承诺时限
  - 工作日 / 自然日
  - 超时处理

☑ 原因码与解释
  - 不通过原因
  - 政策/法规依据

☑ 监管与审计
  - 留痕
  - 统计指标
```

**交互规则**

* 默认全部勾选
* 不允许全部关闭
* 若关闭某项 → 提示「不符合政务场景规范」

---

### 3.2 企业增强模块（Enterprise Enhancement）

#### 显示条件

* scene_type = 企业 · HR / 供应链

---

#### Module Card：HR Enhancement

```
🏢 HR 业务增强
☑ 组织与岗位体系
☑ 编制与岗位约束
☑ 背调与合规校验
☑ 账号与权限开通
☑ 离职与权限回收
```

---

#### Module Card：Supply Chain Enhancement

```
🏭 供应链业务增强
☑ 供应商管理
☑ 库存与物料绑定
☑ 履约与物流跟踪
☑ 财务结算
☑ 异常与兜底策略
```

---

## Section 4：模板约束预览（Template Requirements）

### 🎯 目的

* **提前告诉用户：接下来哪些是“必建模项”**
* 为后面的「模型完整度校验」埋点

---

### Component：TemplateRequirementPreview

**展示形式**

* Tag + 状态

#### 示例（政务 · 一件事）

```
必须完成的建模项：
🟢 核心业务对象
🟢 状态机
🟢 业务规则
🟢 材料 / 数据
🟡 多部门核验
🔴 法定时限（尚未识别）
🔴 原因码（尚未配置）
```

👉 这里的红/黄/绿，会直接映射到后面的：

* 识别结果页
* 建模完整度

---

## Section 5：配置确认区（Actions）

### Component：TemplateConfigActions

#### 按钮

* 「应用模板」
* 「取消」

#### 应用模板后的行为

1. 记录：

   ```
   scene.template_type
   scene.capabilities[]
   ```
2. 驱动：

   * 识别结果页的 **必填校验**
   * 建模页的 **模块显示 / 强制项**

---

# 四、这个 Frame 与你现有页面的“精确衔接点”

### 1️⃣ 对识别结果页的影响

* 业务约束卡：

  * 政务模板 → 默认展开 + 红色强调
* 缺失项：

  * 不再是“AI 没识别到”
  * 而是“模板要求但未满足”

---

### 2️⃣ 对建模页的影响

* 左侧对象列表：

  * 模板决定 **哪些对象必须存在**
* 顶部增加一行：

  ```
  当前模型基于模板：🏛 政务 · 一件事
  ```

---

### 3️⃣ 对“确认模型”的影响

* 未满足模板强制项：

  * 禁用「确认模型」
  * 明确提示缺失原因

---

# 五、你现在立刻可以让设计/前端做的 3 件事
