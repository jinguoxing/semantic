这是一个非常落地的视角。如果我们把数据表看作是一个“容器”，字段看作是“原子”，那么要从这些物理实体中提取出语义，我们需要给它们贴上丰富的**“标签体系”**。

这些属性是构建**数据字典**或**元数据平台**时的物理字段定义。为了方便你的实施，我将属性分为**“表（模型）级”**和**“字段（属性）级”**两个层次来梳理。

---

### 一、 表/模型级属性 (Table/Model Level)

**—— 解决“这堆数据整体是关于什么的”**

这部分属性决定了这张表在业务世界中的定位。

#### 1. 业务身份维 (Business Identity)

* **业务名称 (Business Alias):** 表的中文名。 *例：客户基础信息表*
* **归属域 (Data Domain):** 属于哪个业务域。 *例：CRM域 / 交易域 / 供应链域*
* **对象分类 (Object Type):** 对应我们之前说的五大对象。 *可选值：主体、行为、状态、规则、属性*
* **数据粒度 (Grain/Granularity):** **(最关键)** 一行数据代表什么？ *例：一行代表一个“子订单”，还是一行代表一个“用户的一天”？*

#### 2. 生命周期维 (Lifecycle)

* **数据层级 (Layer):** 在数仓中的位置。 *例：ODS (原始层), DWD (明细层), DWS (汇总层), ADS (应用层)*
* **更新策略 (Update Strategy):** 数据怎么变？ *例：增量追加 (Append)、全量覆盖 (Overwrite)、缓慢变化维 (SCD)*
* **存储周期 (Retention):** 存多久？ *例：永久、3年、7天*

#### 3. 质量与安全维 (Quality & Security)

* **数据负责人 (Data Owner):** 谁对表里的数据准确性负责？（通常是业务方）
* **安全等级 (Security Level):** 整体机密程度。 *例：L1 (公开), L2 (内部), L3 (机密)*

---

### 二、 字段/属性级属性 (Field/Column Level)

**—— 解决“这个具体的格子填的是什么意图”**

这是语义治理的主战场，建议在元数据平台中为字段扩展以下属性列：

#### 1. 核心语义维 (Core Semantics)

* **业务术语 (Business Term):** 对应业务上的哪个指标或属性名称。 *例：将 `c_level` 映射为 `会员等级*`
* **业务口径 (Business Definition):** 一句话解释含义。 *例：指用户注册时的初始等级，非当前动态等级。*
* **主键/标识 (Is Primary/Identifier):** 是否是某个主体的唯一标识 ID。 *例：User ID, Order No*

#### 2. 值域与度量维 (Value & Measure)

* **逻辑数据类型 (Logical Type):** 区别于物理类型（int/varchar）。 *例：金额、枚举、时间点、时间段、手机号、身份证、URL*
* **计量单位 (Unit):** **(数值型字段必填)** *例：元、万元、kg、秒、毫秒、次*
* **字典/枚举引用 (Dict Reference):** **(枚举型字段必填)** 关联到哪个码表。 *例：关联 `dict_order_status` (1=待支付, 2=已支付)*
* **空值含义 (Null Meaning):** NULL 代表什么？ *例：代表“无数据”，还是代表“0”，还是代表“无限大”？*

#### 3. 时间语义维 (Temporal Semantics)

* **时间角色 (Time Role):** 这个时间代表什么环节？ *例：发生时间 (Event Time)、处理时间 (Processing Time)、记录生效时间 (Effective Time)*
* **时区 (Time Zone):** 存储的是 UTC 还是 UTC+8？

#### 4. 关系维 (Relationship)

* **关联外键 (Foreign Key Ref):** 指向哪张表的主键？ *例：`user_id` 关联 `t_user_base.id*`
* **依赖字段 (Dependency):** 它的值依赖谁？ *例：`reject_reason` 依赖于 `status` 字段为 'Rejected' 时才有意义。*

#### 5. 计算逻辑维 (Calculation)

* **是否派生 (Is Derived):** 是原生采集的，还是算出来的？
* **计算公式 (Formula):** 如果是算出来的，逻辑是什么？ *例：`total_price - discount_amount*`
* **来源字段 (Source Mapping):** 对应上游表的哪个字段？（用于血缘分析）

#### 6. 安全隐私维 (Security & Privacy)

* **敏感标签 (Sensitivity Tag):** 是否涉及隐私？ *例：PII (个人隐私), 手机号, 银行卡号*
* **脱敏规则 (Masking Rule):** 展示时如何处理？ *例：中间四位打星*

---

### 三、 总结：你需要构建的“元数据属性清单”

为了方便复制，整理成一份 Checklist：

| 维度 | 属性名 (建议英文名) | 必填性 | 填写示例 |
| --- | --- | --- | --- |
| **表级** | **Business Name** | ★★★ | 客户基础信息表 |
|  | **Object Type** | ★★★ | 主体 (Entity) |
|  | **Grain** | ★★★ | 单个自然人 |
|  | **Layer** | ★★ | DWD |
| **字段级** | **Biz Term** | ★★★ | 注册手机号 |
|  | **Logical Type** | ★★★ | Mobile |
|  | **Unit** | ★★★ | (无) |
|  | **Dict Ref** | ★★ | (无) |
|  | **Calculation** | ★★ | 原值映射 |
|  | **Source Field** | ★★ | ods_user.mobile |
|  | **Sensitivity** | ★ | PII-High |

**实施建议：**
不要试图一开始就让开发人员填满所有属性。

1. **第一阶段（底线）：** 必须填 **业务名称、对象分类、业务口径、单位、枚举引用**。这 5 个是语义理解的最小闭环。
2. **第二阶段（进阶）：** 补充计算公式、来源映射、安全等级。

通过收集这些维度的属性，你的平台就能回答：“这个 `duration` 字段，是**通话时长（业务术语）**，单位是**秒（单位）**，属于**通话行为（对象分类）**，只有在 `status=1` **（枚举含义）**时有效。”
